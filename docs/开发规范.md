# 开发规范

## 开发过程追溯(四层)

为了可靠追踪整个开发过程,采用四层记录体系。**这套体系是工具无关的**:它只规定「记录什么、何时记录、放在哪、如何交叉引用」这个**结果**,不绑定任何特定开发工具(不依赖某个 IDE / Agent 的 hook 或私有会话格式),换工具也照样适用。

四层各司其职,其中 L1 是机械留档,L2/L3/L4 是需要判断的提炼与决策:

| 层 | 记录什么 | 何时记录 | 目录 | 命名 |
|---|---|---|---|---|
| **L1 原始对话** | 一个会话的完整对话记录(问答 + 推理过程 + 工具调用与结果),整理成可读格式、非原始 JSON,不遗漏细节 | 每个会话结束(或告一段落)时 | `docs/开发日志/transcripts/` | `YYYY-MM-DD-<主题或slug>.md`,一个会话一个文件 |
| **L2 精炼开发日志** | 「做了什么、为什么这么做、遇到的问题」的提炼归纳 | 每完成一个开发阶段就追加一小段;一个任务一条 | `docs/开发日志/vibecoding日志/` | `YYYY-MM-DD-<主题>.md` |
| **L3 ADR 决策记录** | 重要架构/技术选择:背景、决策、理由、后果 | 做出重要架构决策时(触发式,不是每次都有) | `docs/开发日志/决策记录/` | `NNNN-<title>.md`(四位序号递增) |
| **L4 提交日志** | 本次提交的改动清单 + commit hash + 关联记录 | 每次向 Git 提交时 | `docs/开发日志/提交日志/` | `YYYY-MM-DD-<commit短hash>-<主题>.md` |

### L1 的导出方式(不绑定工具)

L1 只要求「每个会话结束后,`transcripts/` 下有一份该会话的完整、可读对话记录」这个结果。**具体用什么手段导出,由当前所用的开发工具决定,本规范不写死**——可以是工具自带的导出能力、手动整理、或某个独立脚本。这样规范不会因为更换开发工具而失效。

### 交叉引用规则(以 commit 为锚点)

四层不是孤立的,要能互相追溯,统一以每次提交为锚点串起来:

- **提交日志(L4)** 里注明:commit hash、关联的 vibecoding 日志(L2)、关联的 transcript(L1)、本次涉及的 ADR(L3,若有)。
- **vibecoding 日志(L2)** 里注明对应的提交日志(L4)。
- **ADR(L3)** 里注明关联的 commit、transcript、devlog。
- L1 完整 transcript **随对应代码提交一同入库**(与 commit 对齐),不单独频繁提交。

目录归档的详细约定见 [开发日志/README.md](开发日志/README.md)。

## 测试策略(现阶段:2026-07-04 起)

**现阶段开发新功能时,暂时跳过测试:不要求为新功能编写测试代码,先把功能实现出来,测试留到后续统一补齐。**

- 不需要按 TDD 先写失败测试再实现
- 不需要为新功能补充测试用例
- 例外:如果改动导致**已有测试**断言过时(例如新增工具后,校验工具名单的断言需要同步更新),仍需修正,以保证已有测试套件能跑通——这不算"新写测试",只是维持现状不被破坏

**这是阶段性状态,以后会重新开启"测试先行"的要求。** 到时更新本文档,并在 superpowers 相关的 spec/plan 文档里恢复完整的测试计划。
