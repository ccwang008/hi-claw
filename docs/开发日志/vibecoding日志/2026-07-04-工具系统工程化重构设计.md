# 工具系统工程化重构设计

## 背景

本轮讨论聚焦“现在工具太多,架构如何优化”。当前项目已有 6 个工具,数量本身还可控,但工具实现、schema、handler 注册和确认策略都集中在 `tools.py` 中,继续扩展会让新增工具变得容易漏改、难维护。

## 讨论结论

先不做插件式复杂系统,而是采用中量工程化拆分:

- 新增 `tool_registry.py`,承载 `ToolSpec`、工具注册、schema 生成、统一执行和错误包装。
- 将工具按领域拆成 `tools/` 包,例如 `bash.py`、`files.py`、`search.py`、`list_dir.py`。
- `agent.py` 只保留对话循环、确认交互和日志,不再直接依赖 `TOOLS` / `TOOL_HANDLERS` / `NEEDS_CONFIRMATION`。
- 工具异常应作为 tool_result 返回给模型,避免整轮对话因单个工具错误中断。
- 项目文案从“最简龙虾 Agent”调整为“龙虾 Agent”,定位为可扩展的命令行 AI 编码 Agent。

## 实施结果

已完成工具系统重构:

- 新增 `tool_registry.py`,集中管理 `ToolSpec`、schema 生成、确认策略、默认参数注入和工具错误包装。
- 将工具实现拆分到 `tools/` 包:`bash.py`、`files.py`、`search.py`、`list_dir.py`。
- `tools/__init__.py` 继续兼容旧的 `from tools import ...` 导入,并由 `DEFAULT_REGISTRY` 派生 `TOOLS`、`TOOL_HANDLERS`、`NEEDS_CONFIRMATION`。
- `agent.py` 改为通过 registry 获取 schema 和执行工具,工具异常会作为 tool result 返回,不再直接中断整轮对话。
- README、CLAUDE、AGENTS 中的项目定位文案已从“最简龙虾 Agent”调整为“龙虾 Agent”。

验证:`.venv/bin/python -m pytest -q` 通过,共 42 个测试。
